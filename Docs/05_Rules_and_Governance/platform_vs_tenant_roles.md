## المبدأ الجوهري

النظام يعمل على **محورين مستقلين تماماً**:

```
┌─────────────────────────────────────────────────────────────┐
│                    Platform Layer                           │
│          القائمون على المنصة — يرون البنية والكل           │
├─────────────────────────────────────────────────────────────┤
│                    Tenant Layer                             │
│       مستخدمو الكيان — يرون كيانهم فقط حسب القواعد        │
└─────────────────────────────────────────────────────────────┘
```

**القاعدة الذهبية للفصل:**
> القائم على المنصة لا يملك حق تعديل أي سجل مالي داخل أي كيان.
> المستخدم داخل الكيان لا يرى أي كيان آخر ولا بنية المنصة.

## المحور الأول: القائمون على المنصة (Platform Operators)

### الهرم

```
مالك المنصة (Platform Owner)
        ↓
مشرف المنصة العام (Super Admin)
        ↓
موظف الدعم الفني (Platform Support Agent)
        ↓
محلل البيانات (Platform Analyst)
        ↓
موزّع / شريك (Reseller / Partner)   ← اختياري
```

---

### المستوى 1 — مالك المنصة (Platform Owner)

**من هو:** المطوّر أو الجهة المالكة للمنصة تقنياً وقانونياً.

**ما يملكه:**
- كامل صلاحيات الـ Backend والبنية التحتية
- إنشاء وحذف حسابات المشرفين
- ضبط إعدادات المنصة العامة (خطط الاشتراك، حدود الاستخدام)
- تعطيل أي كيان مخالف أو متوقف

**ما لا يملكه:**
- تعديل السجلات المالية داخل أي كيان
- التصويت أو الموافقة على أي قرار داخل كيان
- الاطلاع على التقارير الطبية أو المستندات الحساسة إلا لسبب قانوني موثق

**ملاحظة:** أي وصول لمالك المنصة لبيانات كيان معين يُسجَّل في Audit Trail خاص مرئي لمدير ذلك الكيان.

---

### المستوى 2 — المشرف العام (Super Admin)

**من هو:** شخص مفوّض من مالك المنصة لإدارة العمليات اليومية.

**ما يراه:**
- لوحة صحة المنصة الكاملة (عدد الكيانات، نشاط الاستخدام، معدلات الخطأ)
- قائمة الكيانات وحالة كل منها (نشط، معلق، مؤرشف)
- بيانات تشغيلية مجمّعة بدون تفاصيل الأفراد

**ما لا يراه:**
- أسماء الأعضاء داخل الكيانات (إلا عند فتح تذكرة دعم مبررة)
- الأرصدة التفصيلية للمحافظ
- المستندات الحساسة

**ما يستطيع فعله:**
- تعليق كيان مخالف لشروط الاستخدام
- تفعيل / إلغاء اشتراكات الكيانات
- رؤية سجلات الأخطاء والأداء

---

### المستوى 3 — موظف الدعم الفني (Support Agent)

**من هو:** موظف يساعد في حل المشكلات التقنية للكيانات.

**وصوله:** مقيّد بالزمن والسبب.

```
Support Access Request {
  entityId        -- أي كيان
  requestedBy     -- موظف الدعم
  reason          -- سبب الوصول
  approvedBy      -- مشرف عام يوافق
  accessWindow    -- مثلاً: 2 ساعة فقط
  loggedAt        -- يظهر في Audit Trail للكيان
}
```

**ما يراه:** البيانات التشغيلية اللازمة لحل المشكلة فقط — لا يرى المستندات الطبية أو الحساسة تلقائياً.

**قاعدة:** كل وصول لموظف الدعم يظهر تلقائياً في لوحة مدير الكيان كإشعار.

---

### المستوى 4 — محلل بيانات المنصة (Platform Analyst)

**من هو:** شخص يحلل أداء المنصة وسلوك الاستخدام لتحسين المنتج.

**ما يراه:** إحصاءات مجمّعة فقط — لا يرى أي بيانات يمكن ربطها بشخص أو كيان بعينه.

```
يرى:   عدد الكيانات النشطة، متوسط حجم المعاملات، نسبة التحصيل الكلية
لا يرى: أسماء، مبالغ تفصيلية، هوية أي عضو
```

---

### المستوى 5 — الموزّع / الشريك (Reseller / Partner) — اختياري

**من هو:** جهة خارجية تُنشئ وتُدير كيانات لعملائها على المنصة.

**ما يملكه:** صلاحيات مؤسس كيان على الكيانات التي أنشأها هو فقط — لا يرى كيانات الآخرين.

**قيد:** لا يملك صلاحيات Super Admin ولا وصولاً للبنية التحتية.

---

## المحور الثاني: مستخدمو الكيان (Tenant Users)

هؤلاء موثّقون بالتفصيل في [access_control_and_privacy.md](./access_control_and_privacy.md)، لكن فيما يلي الهرم الكامل للمرجعية:

### الهرم

```
مؤسس الكيان (Entity Founder)
        ↓
مدير الكيان (Entity Admin)
        ↓
مراجع (Auditor)
        ↓
أمين الصندوق (Treasurer)
        ↓
عضو لجنة (Committee Member)
        ↓
عضو فعّال (Active Member)
        ↓
داعم فقط (Supporter Only)
        ↓
عضو مشروط (Conditional Member)
        ↓
مهتم (Interested)
        ↓
طالب انضمام (Pending Applicant)
```

---

### ملاحظات جوهرية على هرم الكيان

**1. الدور مرتبط بالسياق، لا بالشخص وحده**

نفس الشخص قد يكون مديراً في كيان العائلة وعضواً عادياً في كيان العمارة ومراجعاً في كيان ثالث. الصلاحيات تتحدد بالمعادلة:

```
الشخص + الكيان + المحفظة + المسار + الدور + قاعدة الشفافية
```

**2. مؤسس الكيان ≠ مالك المنصة**

مؤسس الكيان هو من أنشأ صندوق عائلته أو جيرانه — هو أعلى سلطة داخل كيانه فقط. لا علاقة له بالكيانات الأخرى ولا ببنية المنصة.

**3. المراجع: استقلالية حقيقية**

المراجع داخل الكيان يجب أن يكون مستقلاً عن من يراجعهم. لا يملك صلاحية تعديل أي سجل — يملك صلاحية الاطلاع والتنبيه فقط.

```
المراجع يرى:   كل العمليات المالية، المستندات، القرارات، تعارض المصالح
المراجع لا يفعل: لا يوافق، لا يصرف، لا يعدّل، لا يحذف
```

**4. أمين الصندوق: تنفيذ لا قرار**

أمين الصندوق ينفذ القرارات المعتمدة ويوثق الدفع — لا يملك قرار الصرف بنفسه.

```
من يملك المال (أمين الصندوق)  ≠  من يملك قرار الصرف (اللجنة أو الحوكمة)
```

## قواعد الفصل بين المحورين

### 1. لا تداخل في الصلاحيات

| ما لا يُسمح به |
|---|
| موظف دعم المنصة يوافق على صرف داخل كيان |
| مدير كيان يرى بيانات كيان آخر |
| محلل المنصة يرى أسماء الأعضاء |
| مالك المنصة يعدّل سجل مالي لكيان |

### 2. Audit Trail مزدوج

لكل وصول من طبقة المنصة لبيانات طبقة الكيان:

```
PlatformAccessLog {
  platformUserId      -- من المنصة
  entityId            -- أي كيان
  accessType          -- read / support / admin-action
  reason              -- السبب الموثق
  dataScope           -- ما الذي وصل إليه
  accessedAt
  expiredAt
  notifiedEntityAdmin -- هل أُشعر مدير الكيان؟ (true/false)
}
```

هذا السجل **مرئي لمدير الكيان** ليعرف متى وصل أحد من المنصة لبياناته.

### 3. الفصل منطقي أولاً — وليس بالضرورة قواعد بيانات منفصلة

**في MVP (القرار المعتمد):** قاعدة PostgreSQL واحدة مشتركة، فصل منطقي عبر السياق.

```
-- جداول Platform Layer (منفصلة منطقياً، بدون entity_id)
platform_accounts       -- حسابات القائمين على المنصة
platform_roles          -- owner / super_admin / support / analyst
platform_access_logs    -- كل وصول من المنصة لبيانات كيان
platform_subscriptions  -- اشتراكات الكيانات في المنصة (الخطط)

-- جداول Tenant Layer (كل جدول يحمل entity_id كسياق)
entities, members, wallets, transactions, documents, audit_trail
-- كل صف يحمل entity_id → PBAC في التطبيق يضمن العزل
```

**لماذا هذا كافٍ في MVP:**
- PBAC في التطبيق يمنع مدير كيان من رؤية بيانات كيان آخر
- `platform_accounts` لا تحمل `entity_id` → لا يمكن الخلط
- `PlatformAccessLog` يسجّل كل تجاوز للحدود

**لاحقاً (Enterprise / نشر خاص):**
- إمكانية فصل schema مستقل لكيانات كبيرة
- إمكانية RLS (Row Level Security) على مستوى PostgreSQL
- إمكانية قاعدة بيانات مستقلة لعملاء يشترطون العزل الكامل

```
MVP Now:       Single DB → Logical separation via entity_id + PBAC
Later:         Single DB → RLS
Enterprise:    Separate schema or DB per tenant (if needed)
```

### 4. Break-Glass Access — الوصول الاستثنائي الطارئ

لحالات الطوارئ التقنية أو الأمنية أو الأوامر القانونية، يُسمح لمالك المنصة بوصول استثنائي خارج الإجراء المعتاد.

**شروط التفعيل:** طارئ أمني موثق، أو أمر قانوني، أو عطل حرج يمنع التواصل الطبيعي مع مدير الكيان.

```
BreakGlassAccessLog {
  platformOwnerId
  entityId
  triggerReason       -- "security incident" / "legal order" / "critical bug"
  dataAccessed        -- وصف دقيق لما تم الوصول إليه
  startedAt
  endedAt
  notifiedEntityAdmin -- إشعار فوري عند انتهاء الوصول
  reviewScheduledAt   -- مراجعة لاحقة إلزامية
}
```

**القاعدة:** كل وصول Break-Glass يُشعَر به مدير الكيان فور انتهائه، ويُراجَع داخلياً خلال 48 ساعة.
لا يُستخدم Break-Glass لأغراض الدعم العادي — له مسار Support Access منفصل.

---

### 5. انتحال الجلسة ممنوع (No Impersonation)

لا يجوز لأي موظف في المنصة الدخول بجلسة عضو داخل كيان.

```
مسموح:   Support Agent يفتح "Support View" للكيان — للقراءة فقط مع وسم واضح
ممنوع:   Support Agent يبدأ جلسة كأنه "أحمد العضو" ويتصرف بدلاً عنه
```

**واجهة Support View:**
- شريط تنبيه ثابت في أعلى الشاشة: `⚠ وضع عرض الدعم — قراءة فقط — [اسم الموظف] — [وقت الوصول]`
- كل الأزرار التفاعلية معطّلة
- لا يمكن رفع مستندات أو التصويت أو الموافقة
- ينتهي تلقائياً عند انتهاء `accessWindow`

## تأثير هذا التمييز على الواجهة

### لوحتان رئيسيتان منفصلتان تماماً

| اللوحة | لمن | ما تعرضه |
|---|---|---|
| **Platform Dashboard** | المشرف العام، موظف الدعم | صحة المنصة، الكيانات، الاشتراكات، الأخطاء |
| **Entity Dashboard** | مؤسس الكيان، مدير الكيان | حوكمة الكيان، المحافظ، المراجعات |

**مركز صحة الصندوق** ([fund_health_center.md](../04_User_Interfaces/fund_health_center.md)) يظهر في كلا اللوحتين لكن بنطاق مختلف:
- **في Entity Dashboard:** صحة كيان واحد فقط — مؤشرات محافظه ومساراته
- **في Platform Dashboard:** صحة كل الكيانات مجمّعة — مؤشرات صحة المنصة ككل

### الروابط اللوجستية

```
Platform Login URL  →  Platform Dashboard  (مشرف / دعم / محلل)
Tenant Login URL    →  Entity Dashboard    (مدير / عضو / مراجع)
```

المستخدم لا يرى سوى بوابة دخوله. لا يوجد "تبديل" بين البوابتين إلا لمن يملك صلاحيات في المحورين (نادر جداً — مثل المطوّر الذي هو أيضاً عضو في صندوق عائلته).

### صفحة "وصول المنصة إلى بياناتك" في واجهة مدير الكيان

مدير الكيان يرى صفحة مخصصة تُترجم `PlatformAccessLog` بلغة واضحة:

```
┌──────────────────────────────────────────────────────┐
│  وصول فريق المنصة إلى بيانات كيانك                  │
├──────────────────────────────────────────────────────┤
│  موظف الدعم: خالد م.                                │
│  السبب: حل مشكلة تقنية في رفع المستندات             │
│  البيانات التي شاهدها: سجلات العمليات               │
│  وقت الوصول: 14:30 — 15:45 (اليوم)                  │
│  الحالة: انتهى ✓                                     │
├──────────────────────────────────────────────────────┤
│  لم يحدث أي وصول آخر خلال آخر 90 يوماً             │
└──────────────────────────────────────────────────────┘
```

**ما تعرضه الصفحة:**
- من دخل (الاسم والدور في المنصة)
- متى (وقت البداية والنهاية)
- لماذا (السبب المسجّل)
- ماذا شاهد (نطاق البيانات)
- هل انتهت صلاحية الوصول أم لا

**ما لا تعرضه:** تفاصيل تقنية داخلية لا تعني مدير الكيان.

### واجهة إنشاء الكيان: من أين تبدأ؟

مستخدم يريد إنشاء كيان جديد يمر بخطوة على مستوى المنصة أولاً:

```
1. [Platform Layer] — هل عندك اشتراك يسمح بإنشاء كيان؟
   ↓ نعم
2. [Platform Layer] — هل الصندوق خاص أم عام؟ (تحذير قانوني)
   ↓ يختار
3. [Tenant Layer] — معالج إنشاء الكيان (7 خطوات في admin_dashboard.md)
```

هذا يحلّ غياب التحذير القانوني من [LEGAL_AND_COMPLIANCE_BOUNDARIES.md](../06_Optional_Policy/LEGAL_AND_COMPLIANCE_BOUNDARIES.md) — التحذير يأتي في الخطوة 2 على مستوى المنصة، قبل دخول معالج إنشاء الكيان.

## MVP

في النسخة الأولى، يكفي دعم:

**Platform Layer (بسيط جداً):**
- مالك المنصة واحد (مطوّر المشروع نفسه)
- لا يوجد Super Admin منفصل بعد
- لا يوجد Support Agent منفصل بعد
- لوحة بسيطة: قائمة الكيانات + حالتها + عدد الأعضاء

**Tenant Layer (كامل):**
- كل المستويات من مؤسس الكيان حتى طالب الانضمام
- PBAC كما هو موثق في access_control_and_privacy.md

**التمييز التقني في MVP:**
- مسار تسجيل دخول واحد، لكن النظام يعرف هل هذا platform_user أو tenant_user
- PlatformAccessLog موجود من البداية — لا يُضاف لاحقاً

## سياسة تعليق الكيان (Platform Suspension Policy)

بما أن مالك المنصة / المشرف العام يستطيع تعليق كيان مخالف، يجب تحديد أثر التعليق بدقة لأن تعليق صندوق عائلة أو جيران قرار حساس اجتماعياً.

### أنواع التعليق

| النوع | المعنى | الأثر |
|---|---|---|
| `OperationalSuspension` | إيقاف العمليات الجديدة | لا صرف / لا اشتراكات جديدة، لكن الدخول مسموح |
| `FullSuspension` | إيقاف كامل | لا دخول للأعضاء، لا عمليات |
| `ReadOnlySuspension` | قراءة فقط | الأعضاء يرون بياناتهم لكن لا يتصرفون |
| `PendingReview` | قيد المراجعة | الكيان يعمل لكن بإشعار واضح للمدير |

### قواعد التعليق

1. **التعليق لا يحذف البيانات** — الأرصدة والسجلات تبقى كما هي.
2. **يجب إشعار مدير الكيان قبل التعليق** بمدة كافية إلا في حالات الطوارئ الأمنية.
3. **يجب إظهار سبب التعليق** لمدير الكيان — لا تعليق صامت.
4. **تصدير البيانات مسموح دائماً** حتى أثناء التعليق — الكيان يملك بياناته.
5. **حق الاعتراض:** مدير الكيان يستطيع رفع اعتراض رسمي على قرار التعليق.
6. **النزاعات المفتوحة تبقى** — التعليق لا يلغي حق الأعضاء في استكمال اعتراضاتهم القائمة.
