# قرار اعتماد بيئة الاستضافة والتشغيل للإنتاج

بناءً على مراجعة متطلبات تشغيل النظام في بيئة الإنتاج، وبناءً على قرارات المشروع المتعلقة بتقليل التكلفة وعدم الاعتماد على خدمات خارجية مدفوعة، تم اعتماد سيرفر **Hostinger VPS KVM 2** كبيئة الإنتاج الأساسية والدائمة للنظام (ولا مجال للترقية إلى Kubernetes لاحقاً كقرار حاسم).

## أولاً: بيانات السيرفر المعتمد

تم اعتماد السيرفر الحالي بالمواصفات التالية:

```text
Provider: Hostinger
Plan: VPS KVM 2
Location: Netherlands - Meppel
Operating System: Ubuntu 24.04 LTS
CPU: 2 Cores
Memory: 8 GB RAM
Disk Space: 100 GB
Bandwidth: 8 TB
Backup Schedule: Weekly
```

## ثانياً: سبب الاعتماد

تم اختيار هذا السيرفر لأنه يحقق متطلبات المرحلة الحالية من المشروع من حيث:

* تشغيل النظام بالكامل على سيرفر واحد.
* دعم Docker و Docker Compose.
* توفر ذاكرة 8GB RAM، وهي مناسبة لتشغيل خدمات النظام مجتمعة.
* توفر 2 CPU Cores، وهي كافية كبداية للإنتاج الخفيف إلى المتوسط.
* توفر مساحة 100GB مناسبة كبداية لتخزين قاعدة البيانات والمرفقات محلياً.
* تقليل التكلفة التشغيلية وعدم الحاجة إلى خدمات خارجية إضافية في المرحلة الأولى.

## ثالثاً: الخدمات التي سيتم تشغيلها على السيرفر

يتم تشغيل الخدمات التالية على نفس السيرفر باستخدام Docker Compose:

```text
Caddy
Frontend - Next.js
Backend - NestJS
Postgres
Redis
Local Uploads Storage
Local Backups
OpenSearch
Temporal Workflow Engine
```

ويتم اعتماد Caddy كخادم وكيل لإدارة الوصول الخارجي وشهادات HTTPS.

## رابعاً: قرار التخزين

نظراً لرغبة المشروع في عدم استخدام خدمات خارجية مثل S3 أو DigitalOcean Spaces أو Cloudflare R2، يتم اعتماد التخزين المحلي للمرفقات داخل السيرفر.

المسار المقترح للمرفقات:

```text
/opt/collective-trust/uploads
```

ويجب ربط هذا المسار بحاوية الباك إند عبر Docker volume أو bind mount لضمان بقاء الملفات بعد إعادة تشغيل أو إعادة بناء الحاويات.

## خامساً: قرار النسخ الاحتياطي

رغم توفر نسخ احتياطي أسبوعي من مزود الاستضافة، لا يتم الاعتماد عليه وحده.

يتم اعتماد الآتي:

1. نسخة احتياطية يومية من قاعدة بيانات Postgres.
2. نسخة احتياطية يومية من مجلد المرفقات.
3. حفظ النسخ اليومية داخل مجلد واضح في السيرفر.
4. تنزيل نسخة أسبوعية يدوياً خارج السيرفر إلى جهاز المالك أو وسيط تخزين خارجي.

المسار المقترح للنسخ الاحتياطي:

```text
/opt/collective-trust/backups
```

ويعتبر النسخ الاحتياطي اليومي شرطاً أساسياً قبل اعتبار البيئة جاهزة للإنتاج.

تم اعتماد سكربتات تشغيل مباشرة مع النشر:

```text
/opt/collective-trust/backup.sh            نسخة قاعدة البيانات والمرفقات
/opt/collective-trust/restore.sh           استعادة محروسة بتأكيد صريح
/opt/collective-trust/production-smoke.sh  فحص إنتاج سريع بعد النشر
```

يشترط اختبار الاستعادة دورياً على بيئة staging أو نسخة معزولة قبل الاعتماد على أي backup كنسخة صالحة.

## سادساً: قرار الشبكة والأمان

يتم فتح المنافذ التالية فقط للعامة:

```text
80
443
```

ولا يتم كشف المنافذ الداخلية التالية على الإنترنت:

```text
5432  Postgres
6379  Redis
7233  Temporal
9200  OpenSearch
3000  Frontend internal
3001  Backend internal
```

ويكون الوصول إلى النظام من الخارج عبر Caddy فقط.

## سابعاً: قرار Docker

يتم اعتماد Docker و Docker Compose كطريقة التشغيل الرسمية في الإنتاج، ولا يتم تشغيل النظام يدوياً مباشرة على السيرفر.

سبب القرار:

* إعدادات الإنتاج الحالية مبنية على Docker Compose.
* Docker يقلل أخطاء الإعداد اليدوي.
* يسهل نقل النظام مستقبلاً إلى سيرفر آخر.
* لا يضيف تكلفة تشغيلية مؤثرة.
* يناسب تشغيل جميع الخدمات على سيرفر واحد.

## ثامناً: حدود المرحلة الحالية

لا يتم في هذه المرحلة استخدام:

```text
Managed Database
Managed Redis
Load Balancer
CDN
Object Storage خارجي
```

> **ملاحظة هامة:** بناءً على التحديثات الأخيرة لقرارات النطاق، تم استبعاد خيارات مثل (S3, Kubernetes) نهائياً من خطط التطوير المستقبلية، واعتماد الـ VPS والتخزين المحلي كقرارات استراتيجية نهائية.

## تاسعاً: فحص ما بعد النشر

بعد كل نشر إنتاجي، يتم تشغيل `production-smoke.sh` للتحقق من:

1. وصول الصفحة العامة عبر Caddy.
2. وصول `/health` العام إلى الباك إند.
3. صحة الباك إند داخلياً.
4. توفر Swagger JSON داخلياً.
5. تعطيل دخول المطورين في production.
6. صحة OpenSearch داخلياً.
7. صحة Temporal داخلياً.

## عاشراً: قرار الترقية

لا توجد حاجة حالياً للترقية إلى خطة أعلى من KVM 2.

يتم النظر في الترقية لاحقاً فقط عند ظهور أحد المؤشرات التالية:

* زيادة واضحة في عدد المستخدمين.
* امتلاء مساحة التخزين.
* بطء متكرر في النظام.
* ارتفاع استهلاك الذاكرة أو المعالج.
* زيادة كبيرة في حجم المرفقات.
* ضغط كبير على قاعدة البيانات أو التقارير.

## القرار النهائي

تم اعتماد سيرفر **Hostinger VPS KVM 2** كبيئة الإنتاج الأولى للنظام، وتشغيل النظام عليه باستخدام **Ubuntu 24.04 LTS + Docker Compose**، مع تشغيل جميع خدمات النظام على نفس السيرفر، واعتماد التخزين المحلي للمرفقات، والنسخ الاحتياطي اليومي الداخلي، بالإضافة إلى النسخ الأسبوعي المتوفر من مزود الاستضافة.

ويعتبر هذا القرار مناسباً للمرحلة الحالية لأنه يحقق توازناً بين:

* تقليل التكلفة.
* بساطة التشغيل.
* عدم الاعتماد على خدمات خارجية.
* قابلية النقل لاحقاً.
* موارد كافية كبداية للإنتاج.
