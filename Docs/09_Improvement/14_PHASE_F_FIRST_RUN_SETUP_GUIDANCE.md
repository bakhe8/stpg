# 14 - Phase F First-run Setup Guidance

## حالة الوثيقة

الحالة: F-002 منفذة ومتحقق منها.

التاريخ: 2026-07-01.

الإصدار التشغيلي: 2.18.

## الهدف

تهذيب تجربة ما بعد إنشاء الصندوق بدون إعادة فتح نموذج الإنشاء أو تغيير قاعدة البيانات.

الهدف المنتج:

> المؤسس لا يحتاج أن يفهم كل التفاصيل وقت الإنشاء. يستطيع إنشاء صندوق خفيف، ثم يرى من داخل الصندوق ما الخطوة التالية بوضوح.

## القرار التنفيذي

تم تنفيذ F-002 كتغيير واجهي محدود:

- عند إنشاء صندوق من المسار الجديد، ينتقل المستخدم إلى صفحة الصندوق مع إشارة `created=1`.
- إذا بدأ بدون قالب، تحمل الصفحة `start=empty`.
- إذا بدأ من قالب، تحمل الصفحة `start=template`.
- صفحة الصندوق تستخدم هذه الإشارة لإظهار إرشاد أول تشغيل فوق checklist التشغيل.
- checklist نفسها بقيت مرتبطة بالحقائق التشغيلية الموجودة: الوصف، البنك، المحفظة، مسار الحوكمة، والأعضاء.

لا يوجد تغيير في:

- schema.
- API contract.
- routes العامة.
- `Entity` أو `EntityType`.
- صلاحيات الوصول.
- منطق القوالب أو المحافظ أو المسارات.

## ما تغير في الواجهة

### مسار إنشاء الصندوق

الملف: `frontend/src/app/(main)/entities/new/page.tsx`.

بعد إنشاء الصندوق، لم يعد الانتقال إلى:

`/entities/:id`

بل إلى:

`/entities/:id?created=1&start=empty`

أو:

`/entities/:id?created=1&start=template`

هذه الإشارة لا تنشئ حالة دائمة في قاعدة البيانات. هي فقط سياق واجهي للجلسة الأولى بعد الإنشاء.

### صفحة تفاصيل الصندوق

الملفات:

- `frontend/src/app/(main)/entities/[id]/page.tsx`
- `frontend/src/app/(main)/entities/[id]/entity-detail.module.css`

عند وجود `created=1` تظهر لوحة أول تشغيل داخل setup checklist.

إذا كان المسار `start=empty`، تظهر الرسالة:

- الصندوق الفارغ مقصود وليس نقصا.
- تم إنشاء هيكل خفيف.
- التوسع بالمحافظ والمسارات والقواعد يتم من داخل الصندوق.

إذا كان المسار `start=template`، تظهر الرسالة:

- القالب نقطة بداية جاهزة.
- كل ما أنشأه القالب قابل للمراجعة والتعديل.
- المحافظ والمسارات والقواعد لا تزال قابلة للتعديل من داخل الصندوق.

### الخطوة التالية

اللوحة تعرض أول بند ناقص في checklist كخطوة مقترحة:

- إذا لا توجد محفظة: `أول محفظة`.
- إذا توجد محفظة ولا يوجد مسار: `مسار الحوكمة`.
- إذا لا يوجد وصف أو بنك: تقود للإعدادات.
- إذا بقيت دعوة الأعضاء: تقود لنسخ رابط الدعوة.

هذا يحافظ على العمق الحالي، لكنه يخفض عبء القرار في شاشة الإنشاء.

## ما بقي كما هو

- `ابدأ فارغا` لا يمنع إنشاء محافظ لاحقا.
- القالب لا يقفل الصندوق على نموذج تشغيل واحد.
- المؤسس يستطيع الوصول للإعدادات، المحافظ، المسارات، الأعضاء، والقواعد بعد الإنشاء.
- الحملة لم تتغير في هذا البند؛ lifecycle الحملة يبقى منفصلا.
- النموذج القديم لا يزال متاحا عبر rollback flag.

## أثر المنتج

### على المؤسس

قبل F-002:

- بعد إنشاء صندوق فارغ، يرى checklist عامة.
- قد يفسر الفراغ كحالة ناقصة أو خطأ.

بعد F-002:

- يرى أن الصندوق الفارغ اختيار مقصود.
- يرى أول خطوة عملية مباشرة.
- لا يحتاج لاختيار عائلة/عمارة/حي أو قالب حتى يبدأ.

### على الفريق

- صار عندنا فرق واضح بين `نقطة البداية` و`قدرات الصندوق`.
- يمكن لاحقا تحسين F-003 وF-004 فوق نفس المنطق بدون كسر.
- لا يوجد دين migration أو rename.

## المخاطر المحكومة

| الخطر | المعالجة |
|---|---|
| أن يظن المستخدم أن الصندوق الفارغ معطل | أضيفت رسالة أول تشغيل تؤكد أنه مقصود |
| أن تزدحم شاشة الإنشاء بتفاصيل إضافية | لم نضف حقولا جديدة في الإنشاء |
| أن تخفي القوالب أو `ابدأ فارغا` قدرات متقدمة | بقيت القدرات داخل صفحة الصندوق والمحافظ والمسارات |
| أن يتأثر rollback | لم يتغير الفلاق القديم |
| أن يتأثر API أو schema | لا يوجد أي تعديل backend أو migration |
