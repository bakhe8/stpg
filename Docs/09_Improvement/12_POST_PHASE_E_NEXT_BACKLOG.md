# 12 - Post Phase E Next Backlog

## حالة الوثيقة

هذه وثيقة ترشيح للعمل التالي بعد إغلاق Phase E.

الحالة: F-001 منفذة ومتحقق منها؛ المتبقي يبدأ من F-002.

التاريخ: 2026-07-01.

## لماذا هذه الخطوة؟

Phase E أغلقت تنظيف لغة المنتج: المستخدم يرى `صندوق` أو `حملة`، بينما بقيت `Entity` والعقود الداخلية كما هي.

الخطوة التالية لا يجب أن تكون إعادة تسمية داخلية أو route alias. القيمة الأعلى الآن هي التأكد أن تجربة الصندوق الافتراضية صارت سهلة للمؤسس والعضو، وأن كل القدرات العميقة التي بنيناها في 08 و09 ما زالت قابلة للوصول بعد التبسيط.

القاعدة:

> نثبت سهولة التجربة الجديدة بدون أن نحذف أو نخفي العمق التشغيلي.

## خط الأساس المعتمد

| المجال | القرار الحالي |
|---|---|
| Production readiness 08 | مغلق حسب `Docs/08_Production_Readiness/BACKLOG.md` |
| Improvement 09 | Phase A إلى Phase E مغلقة |
| تجربة الإنشاء | مسار صندوق/حملة هو default |
| rollback | `NEXT_PUBLIC_ENABLE_FUND_CREATE_FLOW=false` يبقي النموذج القديم مؤقتا |
| المصطلحات الظاهرة | صندوق / حملة |
| المصطلحات الداخلية | `Entity`, `EntityType`, `/entities`, `X-Entity-ID` باقية |
| قاعدة البيانات وAPI | لا تغيير بعد Phase E |
| route alias | لا نضيف `/funds` الآن |

## قرارات المنتج المثبتة

هذه القرارات لا يعاد فتحها في هذا backlog إلا إذا ظهر سبب منتج واضح:

- الاسم الظاهر للمستخدم هو `صندوق`.
- الحملة تبقى خيارا ظاهرا عندما يكون لها lifecycle مؤقت أو صندوق أب.
- التصنيف الاجتماعي مثل عائلة/عمارة/حي/قبيلة اختيار ثانوي أو profile، وليس شرطا لإنشاء الصندوق.
- القالب نقطة بداية تشغيلية، وليس قيدا على قدرات الصندوق.
- `ابدأ فارغا` خيار مهم ويجب أن يبقى قابلا للتوسع بعد الإنشاء.
- المؤسس يملك الإعدادات المتقدمة، ويمكن تفويضها لاحقا لعضو مناسب.
- تعدد المحافظ لا يحتاج سؤالا في الإنشاء؛ يمكن إنشاء محافظ إضافية من داخل الصندوق.
- الشاشة القديمة تبقى rollback مؤقتا إلى أن تثبت التجربة الجديدة أكثر.
- لا نحذف `Entity` أو `EntityType` أو `/entities` في هذه المرحلة.

## ما لا نبدأ به

هذه البنود مؤجلة عمدا حتى لا نفتح تغيير جذري بدون حاجة:

- إعادة تسمية `Entity` في الكود أو قاعدة البيانات.
- إضافة `/funds` كـ alias أو redirect.
- حذف النموذج القديم أو rollback flag.
- نقل routes أو headers أو DTOs إلى أسماء جديدة.
- migration لقاعدة البيانات من أجل المصطلحات فقط.
- فرض قالب أو profile يمنع الصندوق من الوصول إلى قدرات أخرى لاحقا.

## الترشيح التنفيذي: Phase F

الاسم المقترح: `Phase F - Post-default UX and Capability Hardening`.

هدف Phase F أن تثبت التجربة الجديدة بعد أن أصبحت default، لا أن تضيف مفهوما جديدا.

### F-001 - Capability Preservation Evidence Pack

الحالة: منفذة ومتحقق منها في `13_PHASE_F_CAPABILITY_EVIDENCE_PACK.md`.

الأولوية: P0.

الهدف: تحويل جرد القدرات في `02_CAPABILITY_PRESERVATION_AUDIT.md` إلى checklist تحقق عملي مختصر بعد default switch.

النطاق:

- إنشاء صندوق من كل قالب موجود.
- إنشاء صندوق فارغ ثم الوصول إلى setup checklist.
- إنشاء حملة من صندوق أب.
- إضافة محفظة بعد الإنشاء.
- إضافة مسار حوكمة بعد الإنشاء.
- دعوة عضو أو قبول طلب انضمام.
- إنشاء اشتراك أو مستحق أو إثبات دفع.
- إنشاء قرار وتصويت وتنفيذ أو متابعة حالة التنفيذ.
- إنشاء لجنة وربطها بمسار عند الحاجة.
- فتح اعتراض أو نزاع.
- التأكد أن التدقيق وسطح العمل وصحة الصندوق لا تزال تظهر السجل بلغة صندوق/حملة.

معيار الإغلاق:

- ملف evidence مختصر داخل `Docs/09_Improvement` أو تحديث هذه الوثيقة بنتيجة الفحص.
- أي فجوة تتحول إلى بند صغير قبل الانتقال إلى تحسينات واجهية.
- لا تعديل schema.

نتيجة التنفيذ:

- أضيف ملف evidence مستقل.
- مرت اختبارات backend targeted، وPhase D create-flow smoke، وUX role audit، وseed validation.
- لا توجد فجوة blocking؛ ينتقل العمل التالي إلى F-002.

### F-002 - First-run Setup Guidance Polish

الأولوية: P1 بعد F-001.

الهدف: جعل ما بعد إنشاء الصندوق واضحا بدون إجبار المؤسس على فهم تفاصيل تقنية في شاشة الإنشاء.

النطاق:

- مراجعة checklist بعد إنشاء الصندوق.
- التأكد أن الروابط تقود إلى المكان الصحيح: الإعدادات، المحافظ، المسارات، الأعضاء، القواعد.
- التأكد أن `ابدأ فارغا` لا يبدو كحالة معطلة أو ناقصة بشكل مربك.
- مراجعة copy الصندوق والحملة في first-run فقط.

معيار الإغلاق:

- المؤسس يعرف ما الخطوة التالية من داخل الصندوق.
- لا تظهر إعدادات متقدمة في البداية إلا عند الحاجة أو من رابط واضح.
- لا يوجد نص يعيد `كيان` للمستخدم.

### F-003 - Advanced Settings Access Audit

الأولوية: P1.

الهدف: التأكد أن العمق التشغيلي موجود، لكنه موضوع في المكان الصحيح وللأدوار الصحيحة.

النطاق:

- المؤسس يرى الإعدادات المتقدمة.
- العضو المفوض عبر `canManageAdvancedSettings` يرى ما يلزمه.
- أمين الصندوق والمراجع وعضو اللجنة لا يحصلون تلقائيا على صلاحيات policy عامة بسبب دورهم.
- تغييرات السياسة الحساسة تظل مؤرخة في audit ولها impact واضح.

معيار الإغلاق:

- لا يوجد خلط بين أدوار التشغيل وصلاحية الإعدادات المتقدمة.
- أي نقص في الواجهة أو الحماية يتحول إلى بند تنفيذ مستقل.

### F-004 - Template Capability Matrix

الأولوية: P2.

الهدف: تثبيت أن القوالب الحالية نقاط بداية فقط.

النطاق:

- `ابدأ فارغا`.
- صندوق مرن.
- تكافل ومساعدات.
- خدمات مشتركة.
- دعم وتبرعات.
- حملة مرتبطة بصندوق أب.

لكل قالب نثبت:

- ما الذي ينشأ مباشرة.
- ما الذي يستطيع المؤسس إضافته لاحقا.
- ما القدرات التي لا يجب أن يحجبها القالب.
- هل توجد محفظة `SHARED` أو `SEPARABLE` حيث يلزم.

معيار الإغلاق:

- مصفوفة واضحة للفريق والاختبارات.
- لا يوجد قالب يقفل الصندوق داخل نموذج واحد.

### F-005 - Legacy and URL Hygiene Watch

الأولوية: P2.

الهدف: إبقاء المسار القديم و`/entities` مستقرين بدون أن يربكا المستخدم.

النطاق:

- التأكد أن rollback flag يعمل عند الحاجة.
- إبقاء الروابط الحالية والاختبارات القائمة مستقرة.
- عدم إضافة `/funds` إلا إذا ظهر احتياج منتج واضح.
- تسجيل أي مكان يضطر المستخدم لرؤية `/entities` بشكل مزعج، بدون تنفيذه كـ alias الآن.

معيار الإغلاق:

- لا يوجد كسر للروابط أو bookmarks.
- قرار `/funds` يبقى مؤجلا ومقيدا بحزمة مستقلة additive إذا احتجناه.

## ترتيب التنفيذ المقترح

1. `F-001` أولا لأنه يثبت عدم خسارة القدرات.
2. `F-002` بعده لأنه يحسن تجربة المؤسس بناء على ما يظهر في الفحص.
3. `F-003` بالتوازي أو بعد F-002 إذا ظهرت فجوات صلاحيات.
4. `F-004` بعد استقرار الأدلة، لتتحول القوالب إلى مرجع منتج واختبار.
5. `F-005` يبقى watch item ولا يتحول إلى تنفيذ إلا إذا ظهر احتياج واضح.

## بوابات التحقق المقترحة

حسب نوع التغيير:

| نوع التغيير | التحقق الأدنى |
|---|---|
| توثيق فقط | `git diff --check` |
| واجهة فقط | `npm run lint` في frontend و`npm run build` عند الحاجة |
| تجربة إنشاء أو labels | `npm run test:phase-d:create-flow` |
| أدوار وسطح يومي | `npm run test:ux:roles` |
| backend أو policy | `npm run build` في backend واختبارات targeted |
| schema أو seed | `npx prisma validate` وseed validation |
| قدرات Phase F | checklist F-001 مع أدلة مختصرة |

## قرارات تحتاج موافقة قبل فتحها

لا تحتاج F-001 موافقة منتج إضافية لأنها تحقق لا يغير السلوك.

هذه القرارات لا تبدأ تلقائيا:

- حذف rollback القديم.
- إضافة `/funds` alias.
- تغيير اسم `Entity` داخليا.
- تغيير schema من أجل المصطلحات.
- جعل template معين يقيد قدرات الصندوق.
- إخفاء صفحة advanced كاملة عن المؤسس.

## الخلاصة العملية

الترشيح هو ألا نبدأ بتغيير كبير جديد.

تم إغلاق `F-001`: ثبت بالأدلة أن المسار الجديد بعد default switch ما زال يصل إلى التصويت، اللجان، المال، المحافظ، المسارات، الاشتراكات، الصرف، التدقيق، النزاعات، والحملات.

العمل التالي هو `F-002`: تحسين first-run والواجهات المتقدمة بناء على الأدلة لا على افتراضات.
