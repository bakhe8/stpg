# Phase G - Product Acceptance and Release Candidate Backlog

## حالة الوثيقة

**الإصدار التشغيلي:** 2.22  
**التاريخ:** 2026-07-02  
**الحالة:** مفتوحة للتنفيذ  
**الهدف:** تحويل ما أنجزناه في 08 و09 إلى فحص قبول منتج كامل من عين المستخدم.

## لماذا Phase G الآن؟

أغلقنا Phase F بعد تثبيت أن تجربة `صندوق / حملة` أصبحت default، وأن القوالب لا تقفل القدرات، وأن rollback و`/entities` مستقران.

الخطوة الصحيحة الآن ليست إضافة ميزة جديدة ولا تغيير تسمية داخلية. الخطوة الصحيحة هي قبول المنتج كرحلة كاملة:

- هل يستطيع المؤسس إنشاء صندوق وفهم ما بعد الإنشاء؟
- هل يستطيع العضو الانضمام والاشتراك والدفع؟
- هل يستطيع أمين الصندوق أو اللجنة أو المراجع أداء عملهم اليومي؟
- هل بقي العمق التشغيلي موجودا بدون أن يظهر للمستخدم كتعقيد مبكر؟
- هل توجد فجوات polish أو blockers قبل اعتبار الحالة release candidate؟

## قواعد الحماية

هذه المرحلة لا تفتح تلقائيا أي تغيير بنيوي:

- لا نحذف `NEXT_PUBLIC_ENABLE_FUND_CREATE_FLOW=false`.
- لا نضيف `/funds` alias.
- لا نعيد تسمية `Entity` أو `EntityType` داخليا.
- لا نغير schema أو migration لمجرد قبول المنتج.
- لا نضيف ميزة جديدة لمعالجة ملاحظة قبول إلا إذا صُنفت كـ blocker وصارت لها بطاقة مستقلة.
- أي تعديل ينتج من Phase G يجب أن يكون صغيرا، واضح السبب، ومربوطا بسيناريو قبول فشل.

## تعريف Release Candidate

نعتبر الحالة release candidate عندما تتحقق الشروط التالية:

- كل رحلات المؤسس والعضو والأدوار التشغيلية تعمل بدون كسر أو شاشة فارغة أو خطأ console جوهري.
- لا توجد فجوة تمنع إنشاء صندوق، إكمال إعداده، الاشتراك فيه، الدفع له، أو اتخاذ قرار داخله.
- لا توجد قدرة موجودة في 08 أصبحت غير قابلة للوصول بسبب تبسيط 09.
- النصوص الظاهرة لا تعيد المستخدم إلى مفهوم "كيان" في واجهة المستخدم العادية.
- أي ملاحظة متبقية مصنفة بوضوح: blocker أو polish أو later.

## Backlog Phase G

### G-001 - Acceptance Inventory and Evidence Harness

الأولوية: P0.

الهدف: تجهيز قائمة قبول تنفيذية واحدة قبل تشغيل الرحلات.

النطاق:

- تثبيت المستخدمين seed الذين سنفحص بهم الرحلات.
- تثبيت روابط البداية لكل رحلة.
- تثبيت الأدلة المطلوبة: screenshots، console errors، API responses عند الحاجة، ونتائج الاختبارات.
- تحديد أين تحفظ أدلة التشغيل المؤقتة حتى لا تتداخل مع الوثائق الدائمة.

مستخدمون seed مرشحون:

| الدور/الحالة | المستخدم |
|---|---|
| مؤسس عائلة | `seed.ahmed.family` |
| مدير/تشغيل عائلي | `seed.sara.family` |
| أمين صندوق/تفويض متقدم محتمل | `seed.nasser.family` |
| مراجع | `seed.layan.audit` |
| لجنة/حالة علاجية | `seed.majed.medical` |
| عضو اجتماعي | `seed.noura.social` |
| عضو متعدد الصناديق | `seed.faisal.overlap` |
| عضو موقوف | `seed.khaled.suspended` |
| مؤسس عمارة | `seed.abdullah.building` |
| مؤسس حي | `seed.yahya.neighborhood` |
| حملة/حالة | `seed.fahad.case` |
| عضو مشروط | `seed.amal.conditional` |
| مؤسس قبيلة | `seed.abdulrahman.tribe` |
| مدير/أمين عمارة | `seed.mona.building` |

معيار الإغلاق:

- جدول الرحلات جاهز، وكل رحلة لها مستخدم بداية ونتيجة قبول واضحة.
- لا نبدأ إصلاحات قبل انتهاء inventory إلا إذا وجد blocker يمنع تشغيل القبول نفسه.

### G-002 - Founder Empty Fund Journey

الأولوية: P0.

الهدف: قبول رحلة المؤسس عندما يبدأ بصندوق فارغ.

السيناريو:

1. تسجيل دخول مؤسس.
2. فتح `/entities/new`.
3. اختيار `صندوق`.
4. اختيار `ابدأ فارغا`.
5. إنشاء صندوق باسم اختبار واضح.
6. الوصول إلى صفحة الصندوق مع `created=1&start=empty`.
7. رؤية لوحة أول تشغيل والخطوة التالية.
8. الانتقال إلى إنشاء محفظة أو إعدادات الصندوق بدون ضياع.

معيار الإغلاق:

- الصندوق الفارغ لا يظهر كخطأ أو نقص مخيف.
- المستخدم يفهم الخطوة التالية.
- كل الروابط الأساسية في first-run تعمل.
- لا يوجد console error جوهري.

### G-003 - Founder Template Fund Journey

الأولوية: P0.

الهدف: قبول رحلة المؤسس عندما يبدأ من قالب.

السيناريو:

1. إنشاء صندوق من كل قالب أساسي أو عينة ممثلة: مخصص، تكافل، خدمات مشتركة، داعمين فقط.
2. التأكد أن القالب أنشأ المحفظة والمسار المتوقعين.
3. التأكد أن المؤسس يستطيع لاحقا الوصول إلى الإعدادات المتقدمة والقواعد والمحافظ.
4. التأكد أن القالب لا يخفي قدرات أخرى مثل إضافة محفظة جديدة أو مسار جديد.

معيار الإغلاق:

- القالب يختصر البداية ولا يحصر الصندوق.
- `SHARED/SEPARABLE` ظاهرة في موضعها الصحيح عند فحص المحافظ.
- لا يوجد قالب يحول الصندوق إلى مسار واحد مغلق.

### G-004 - Campaign Journey

الأولوية: P0.

الهدف: قبول رحلة الحملة كمسار مستقل مرتبط بصندوق أب.

السيناريو:

1. مؤسس لديه صندوق قائم يفتح `/entities/new`.
2. يختار `حملة`.
3. يختار الصندوق الأب.
4. يحدد اسم الحملة وتاريخا اختياريا.
5. ينشئ الحملة ويفتح صفحتها.
6. يتأكد من ظهور لغة حملة لا صندوق.
7. يتأكد أن خريطة تشغيل الحملة تقود إلى محفظة/مسار الحملة عند الحاجة.

معيار الإغلاق:

- لا تختلط الحملة مع قالب الصندوق.
- لا تظهر الحملة كصندوق دائم.
- lifecycle الحملة واضح: صندوق أب، مدة، تجهيز مالي، وإغلاق/قراءة فقط عند اللزوم.

### G-005 - Member Join, Subscription, and Payment Journey

الأولوية: P0.

الهدف: قبول رحلة العضو العادي من الانضمام إلى المشاركة المالية.

السيناريو:

1. عضو يرى صناديقه أو رابط انضمام.
2. يدخل صندوقا.
3. يرى المسارات المناسبة له.
4. يشترك أو يتابع حالة اشتراكه.
5. يرى المستحقات.
6. يرفع أو يتابع سجل دفع.

معيار الإغلاق:

- العضو لا يحتاج فهم `Entity` أو القوالب.
- حقوقه مرتبطة بالمسار والمحفظة، لا بمجرد وجوده في الصندوق.
- حالات `ACTIVE`, `CONDITIONAL`, `SUSPENDED`, `EXITED`, `SUPPORTER_ONLY` لا تكسر الواجهة.

### G-006 - Operational Roles Journey

الأولوية: P0.

الهدف: قبول يوم العمل للأدوار التشغيلية.

النطاق:

- المؤسس يرى مركز المراجعات والإعدادات المطلوبة.
- المدير يرى ما يلزم لإدارة الأعضاء والطلبات.
- أمين الصندوق يرى المالية والمدفوعات بدون صلاحية policy عامة تلقائية.
- عضو اللجنة يرى قراراته ومساراته.
- المراجع يرى سطح التدقيق والمستندات والتقارير.

معيار الإغلاق:

- كل دور يرى أدواته اليومية فقط في السطح الأساسي.
- الأدوات المتقدمة تبقى موجودة عند الحاجة ووفق الصلاحية.
- لا توجد عودة لنصوص "كيان/كيانات" في واجهة المستخدم العادية.

### G-007 - Governance, Audit, and Dispute Journey

الأولوية: P1.

الهدف: قبول العمق التشغيلي الذي لا يظهر في الإنشاء المبسط.

النطاق:

- إنشاء/متابعة قرار.
- تصويت أو موافقة لجنة.
- طلب صرف وتنفيذه أو رفضه.
- مستندات وخصوصية.
- اعتراض أو نزاع.
- audit trail يوضح ما حدث.

معيار الإغلاق:

- المسارات العميقة قابلة للوصول بعد التبسيط.
- لا توجد قدرة من جرد 08/09 مفقودة بسبب تجربة صندوق/حملة.
- السجل الرقابي واضح بما يكفي للتتبع.

### G-008 - Release Candidate Decision

الأولوية: P0 بعد G-001 إلى G-007.

الهدف: اتخاذ قرار واضح بعد تشغيل القبول.

المخرجات الممكنة:

| القرار | المعنى |
|---|---|
| `RC_READY` | لا توجد blockers. نثبت الحالة كمرشح إصدار. |
| `RC_READY_WITH_POLISH` | لا توجد blockers، لكن توجد ملاحظات polish يمكن جدولتها. |
| `RC_BLOCKED` | توجد فجوات تمنع قبول المنتج، وتفتح backlog إصلاح صغير. |

معيار الإغلاق:

- تقرير قبول مختصر يربط كل ملاحظة بسيناريو.
- كل blocker له ملف/مسار/دليل.
- لا يتحول polish إلى refactor واسع.

## بوابات التحقق المقترحة

| نوع الدليل | الأمر أو الإجراء |
|---|---|
| إنشاء صندوق/حملة | `npm run test:phase-d:create-flow` في frontend. |
| أدوار وسطح يومي | `npm run test:ux:roles` في frontend. |
| قدرات backend الأساسية | `npm run test:phase-d:parity` في backend. |
| صحة seed والقوالب | `npm run seed:validate:docker` في backend. |
| تغييرات واجهة | `npm run lint` و`npm run build` في frontend عند وجود تعديل كود. |
| تغييرات backend | `npm run build` واختبارات targeted في backend عند وجود تعديل كود. |
| توثيق فقط | `git diff --check`. |

## طريقة العمل

كل بند من G يجب أن ينتهي بإحدى ثلاث حالات:

- `Accepted`: الرحلة تعمل ولا تحتاج تعديل.
- `Accepted with notes`: الرحلة تعمل وتوجد ملاحظات polish غير حاجبة.
- `Blocked`: الرحلة لا تقبل المنتج، وتحتاج بطاقة إصلاح مستقلة.

إذا ظهر blocker:

1. نسجله في وثيقة القبول أو backlog إصلاح صغير.
2. نحدد هل هو frontend أو backend أو seed/test data أو wording.
3. نصلحه بحزمة صغيرة.
4. نعيد تشغيل الرحلة التي فشلت فقط، ثم بوابة عامة مناسبة.
