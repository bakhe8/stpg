# المحركات الأساسية (Core Engines)

النظام يعمل بـ 8 محركات أساسية. كل محرك مسؤول عن جانب معين من الحوكمة والقرارات. هذا القسم يشرح دور كل محرك ومسؤولياته.

---

## 1. محرك التوافق (Compatibility Engine)
**الدور:** قلب النظام. يقارن شروط العضو مع خصائص الكيان والمحفظة والمسار وبند الصرف، ويقرر حقوق العضو والتزاماته.

**المدخلات:**
- خصائص العضو وتفضيلاته
- خصائص الكيان
- خصائص المحفظة
- خصائص المسار
- خصائص بند الصرف

**المخرجات:**

| السؤال | الجواب |
|---|---|
| هل العضو مؤهل للاشتراك؟ | نعم / لا / مشروط |
| هل يدفع؟ | نعم إذا أصبح فعالاً |
| هل يستفيد؟ | حسب الاشتراك والاستحقاق |
| هل يحق له التصويت؟ | حسب الكيان / المحفظة / المسار |
| هل يحق له الاعتراض؟ | حسب حقه في المال أو القرار |
| هل يرى التفاصيل؟ | حسب الشفافية والخصوصية |
| هل يوجد تعارض مع شروطه؟ | نعم / لا |
| هل تغيرت القواعد؟ | يحتاج موافقة جديدة |

**مثال 1 — غير متوافق:**
```
العضو يشترط: مجلس إدارة
المسار الحالي: قرار فردي
→ النتيجة: غير متوافق
→ الحالة: مشترك مشروط (Conditional)
→ الدفع: لا  |  الاستفادة: لا
```

**مثال 2 — متوافق:**
```
العضو يقبل: قرار فردي بسقف 500 ريال
المسار: قرار فردي بسقف 500 ريال + تقرير شهري
→ النتيجة: متوافق
→ الحالة: مشترك فعال (Active)
→ الدفع: نعم  |  الاستفادة: نعم حسب شروط البند
```

## 2. محرك القواعد (Rules Engine)
**الدور:** إدارة القواعد المتغيرة والقابلة للتكوين بدلاً من كتابتها بشكل جامد في الكود.

**أمثلة قواعد:**
- لا يجوز نقل أموال من محفظة إلى أخرى إلا بتصويت أصحاب المحفظة.
- إذا كان الصرف فوق 5,000 ريال، يحتاج تصويت أصحاب المسار.
- إذا كان البند حساساً، لا تظهر بيانات المستفيد إلا للجنة.
- إذا تغير نموذج الحوكمة، يجب إعادة موافقة الأعضاء المتأثرين.
- إذا كان العضو غير متوافق مع المسار، لا يتحول إلى دافع فعال.

**تسلسل القواعد (من الأعلى إلى الأدنى):**
```
قواعد الكيان
  ↓
قواعد المحفظة
  ↓
قواعد المسار
  ↓
قواعد بند الصرف
  ↓
شروط العضو
```

> **قاعدة مهمة:** المستوى الأدنى لا يخالف الأعلى إلا إذا سمح الأعلى بالاستثناء.

## 3. محرك العلاقات (Relationship Engine)
**الدور:** إدارة العلاقات بين الكيانات ومنع التضارب والتكرار.

**يعالج:**
- عضو في كيانين (تداخل عضوية)
- محفظة مشتركة بين كيانين
- دعم من صندوق إلى صندوق
- تفويض إدارة ودمج
- منع تكرار الدفع لنفس الغرض
- منع خلط أموال العائلة والعمارة
- توضيح من يقرر ومن يستفيد

> لسيناريوهات عملية، انظر: [entity_relationship_scenarios.md](entity_relationship_scenarios.md)

---

## 4. محرك القرارات والتصويت (Decision & Voting Engine)
**الدور:** كل قرار في النظام يكون كائناً مستقلاً قابلاً للتتبع والمراجعة.

**أنواع القرارات:** إنشاء محفظة، إنشاء مسار، صرف مبلغ، تعديل اشتراك، تعديل حوكمة، نقل رصيد، قبول عضو، فتح نزاع، إغلاق محفظة، دمج مسارات.

**أنواع التصويت المدعومة:**

| نوع التصويت | حالة الاستخدام |
|---|---|
| صوت لكل عضو | قرارات عامة |
| صوت لكل أسرة | عائلات |
| صوت لكل مشترك في مسار | قرارات المسار |
| صوت حسب المساهمة | مشاريع خاصة (بحذر) |
| أغلبية بسيطة | قرارات عادية |
| ثلثان | تعديل لائحة |
| تصويت سري | قرارات حساسة |
| موافقة لجنة | قرارات تشغيلية |
| قرار فردي بسقف | خدمات بسيطة |
| قرار عاجل ثم مراجعة | طوارئ |

## 5. محرك الصلاحيات (Policy-Based Access Control)
**الدور:** صلاحيات مبنية على السياسة والسياق، وليس فقط على الدور.

الصلاحية لا تعتمد فقط على "هل هو عضو أو مدير"، بل على:
- عضو في أي كيان؟
- مشترك في أي محفظة وأي مسار؟
- هل دفع؟ هل هو مستفيد؟ هل هو مراجع؟
- هل البند حساس؟ هل البيانات مالية أم شخصية؟
- هل القرار يخص ماله؟

> لتفاصيل مستويات الصلاحيات، انظر: [access_control_and_privacy.md](../05_Rules_and_Governance/access_control_and_privacy.md)

---

## 6. محرك الشفافية والخصوصية (Transparency & Privacy Engine)
**الدور:** تحقيق التوازن بين الشفافية المالية وحماية خصوصية الأفراد.

**مستويات الظهور:**
| المستوى | المعنى |
|---|---|
| `PublicToMembers` | متاح لكل أعضاء الكيان |
| `VisibleToParticipants` | متاح لمشتركي المحفظة/المسار فقط |
| `VisibleToCommittee` | متاح للجنة فقط |
| `VisibleToAuditor` | متاح للمراجع فقط |
| `HiddenSensitive` | مخفي تماماً إلا لمعالج الحالة |
| `AggregatedOnly` | يظهر كإجمالي فقط بدون تفاصيل |

> **القاعدة:** نخفي الإنسان عند الحاجة، ولا نخفي المال.

---

## 7. محرك الاعتراضات والنزاعات (Appeal & Dispute Engine)
**الدور:** إدارة دورة حياة الاعتراض من التقديم إلى القرار النهائي.

**أنواع الإجراءات:** طلب توضيح، اعتراض، مراجعة، تصعيد، نزاع داخلي، شبهة مخالفة لائحة، شبهة قانونية.

**أسباب الاعتراض:** مخالفة اللائحة، تعارض مصالح، نقص مستندات، تجاوز صلاحية، صرف من محفظة خاطئة، مخالفة شروط العضو، تصويت غير صحيح، طلب إيضاح.

> لتفاصيل إدارة النزاعات، انظر: [dispute_management.md](../05_Rules_and_Governance/dispute_management.md)

---

## 8. محرك التوصيات والتحليل (Analytics & Recommendation Engine)
**الدور:** تحليل البيانات واقتراح تحسينات. ليس ضرورياً في النسخة الأولى.

**الوظائف:**
- تحليل رغبات الأعضاء ومعرفة المطلوب والمرفوض
- اقتراح أفضل نموذج حوكمة بناءً على بيانات فعلية
- كشف التكرار بين صندوقين
- كشف المحافظ الضعيفة وإرهاق الدفع
- كشف انخفاض المشاركة وتضارب المصالح
- قياس صحة الصندوق

**مثال:**
```
إذا اخترنا مجلس إدارة → سينضم 70 عضواً
إذا اخترنا قراراً فردياً → سينضم 20 عضواً فقط
إذا اخترنا نموذجاً هجيناً → سينضم 85 عضواً
```

> لمؤشرات الإنذار المبكر، انظر: [fund_health_center.md](../04_User_Interfaces/fund_health_center.md)

## محرك إضافي: محرك القوالب (Template Engine)
**الدور:** توفير قوالب جاهزة حتى لا يبدأ المستخدم من الصفر.

**أمثلة قوالب:**
- صندوق عائلة بسيط
- صندوق جيران خدمات
- صندوق عمارة
- صندوق طوارئ
- محفظة علاج
- حملة حالة واحدة
- محفظة مشتركة بين كيانين
- مسار مجلس إدارة / قرار فردي بسقف / تصويت عام

كل قالب يحتوي قواعد مبدئية يمكن تعديلها حسب حاجة الكيان.
