# البنية البرمجية والتصميم المعياري (Modular Design)

## المبدأ: Modular Monolith أولاً
المشروع معقد منطقياً، لكن في البداية يجب أن يكون **تطبيقاً واحداً منظماً داخلياً** وليس Microservices. هذا يعطي بساطة في التطوير مع تنظيم يسمح بالفصل لاحقاً.

---

## بنية المجلدات (Backend)

```
src/
├── identity/           # الهوية والتسجيل والمصادقة
├── entities/           # الكيانات (عائلة، عمارة، حي)
├── memberships/        # العضويات وتفضيلات الأعضاء
├── wallets/            # المحافظ المالية
├── governance-paths/   # مسارات الحوكمة
├── spending-items/     # بنود الصرف
├── subscriptions/      # الاشتراكات وحالاتها
├── decisions/          # القرارات
├── voting/             # التصويت
├── appeals/            # الاعتراضات
├── disputes/           # النزاعات
├── ledger/             # الدفتر المالي
├── relationships/      # العلاقات بين الكيانات
├── policies/           # السياسات والقواعد
├── documents/          # المرفقات والمستندات
├── notifications/      # الإشعارات
├── reporting/          # التقارير
└── audit/              # سجل التدقيق
```

---

## الوحدات الداخلية (Modules)

| الوحدة | المسؤولية |
|---|---|
| Identity Module | تسجيل، مصادقة، إدارة حسابات |
| Entity Module | إنشاء الكيانات وسياساتها |
| Membership Module | العضويات، التفضيلات، الأدوار |
| Wallet Module | المحافظ وقواعدها |
| Governance Module | مسارات الحوكمة |
| Decision Module | القرارات والتصويت |
| Ledger Module | الدفتر المالي |
| Appeal Module | الاعتراضات والنزاعات |
| Policy Module | محرك القواعد والسياسات |
| Relationship Module | العلاقات بين الكيانات |
| Reporting Module | التقارير المالية والإدارية |
| Notification Module | الإشعارات (بريد، SMS، Push) |

## API: REST أولاً

**للبداية:** REST أفضل لأنه أوضح وأسهل للفريق، خاصة مع عمليات مثل:
- إنشاء كيان، فتح محفظة، التصويت، دفع اشتراك، تقديم اعتراض

**الاقتراح:**
```
REST أولاً + OpenAPI documentation
→ ثم GraphQL فقط إذا ظهرت حاجة حقيقية (مثلاً: لوحات التحكم المعقدة)
```

## محرك القواعد: داخلي أولاً

لا تستخدم محرك قواعد خارجي معقد من البداية. ابدأ بـ **Policy Engine داخلي بـ TypeScript**:

```
policy/
├── compatibility.policy.ts    # توافق العضو مع المسار
├── voting.policy.ts           # قواعد التصويت
├── eligibility.policy.ts      # أهلية الاستفادة
├── transparency.policy.ts     # مستويات الشفافية
└── wallet-transfer.policy.ts  # قواعد نقل الأرصدة
```

**لاحقاً** إذا كبرت القواعد جداً، يمكن نقل جزء منها إلى:
- JSON rules
- DSL داخلي
- محرك مثل Open Policy Agent

> لا تبدأ بأداة ثقيلة من أول يوم.

---

## الذكاء الاصطناعي

**ليس في جوهر القرار المالي.** لا تجعل AI يقرر من يستحق المال.

لكن يمكن استخدامه لاحقاً في:
- تلخيص اللوائح
- شرح سبب عدم توافق العضو
- اقتراح قالب حوكمة
- كشف تشابه بنود
- تلخيص النزاعات وصياغة المحاضر
- تحليل آراء الأعضاء واكتشاف التكرار والتعارض

**القرار يجب أن يبقى:** قواعد + تصويت + حوكمة بشرية موثقة.
