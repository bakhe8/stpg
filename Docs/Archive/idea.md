### Assistant

نعم، صندوق الجيران يدخل ضمن نفس العائلة العامة من الأفكار، لكنه يختلف في هوية المجموعة.

صندوق الجيران يعني غالبًا:
مجموعة سكان حي، شارع، عمارة، استراحة، أو مجمع سكني يجمعون مبلغًا دوريًا أو عند الحاجة لاستخدامه في أمور مشتركة.

أمثلة استخدامه:

الصندوق	من أعضاؤه؟	أمثلة الصرف
صندوق العائلة	أفراد عائلة	علاج، وفاة، زواج، مساعدة قريب
صندوق القبيلة	أبناء قبيلة أو فخذ	ديات، مساعدات كبيرة، مناسبات عامة
صندوق الجيران	سكان حي/شارع/عمارة	صيانة مشتركة، حراسة، إنارة، نظافة، عزاء، مساعدة جار محتاج
صندوق الطوارئ	أي مجموعة	مرض، حادث، وفاة، أزمة مفاجئة
صندوق الجماعة	اسم عام	يشمل أي مجموعة حسب الاتفاق

فـ صندوق الجيران يشبه صندوق العائلة من حيث التعاون، لكنه ليس مبنيًا على القرابة، بل على السكن والجيرة والمصلحة المشتركة.

وقد يكون له نوعان:

1. صندوق خدمات مشتركة
مثل دفع حارس، كاميرات، صيانة بوابة، تنظيف، إنارة، ماء، مناسبات الحي.

2. صندوق تكافل اجتماعي
مثل مساعدة جار مريض، وفاة، حادث، احتياج مفاجئ، أو دعم أسرة متضررة.

الخلاصة:
صندوق الجيران = صندوق جماعة مبني على علاقة الجيرة.
أما صندوق الطوارئ فهو قد يكون داخل صندوق الجيران أو العائلة أو القبيلة، وليس بالضرورة صندوقًا مستقلًا.

4. محرك القواعد

تحتاج محرك قواعد عام:

Rules Engine

لأن القواعد كثيرة ومتغيرة.

بدل أن تكتب كل شيء في الكود بشكل جامد، تجعل القواعد قابلة للتكوين.

أمثلة قواعد:

لا يجوز نقل أموال من محفظة إلى أخرى إلا بتصويت أصحاب المحفظة.

إذا كان الصرف فوق 5000 ريال، يحتاج تصويت أصحاب المسار.

إذا كان البند حساسًا، لا تظهر بيانات المستفيد إلا للجنة.

إذا تغير نموذج الحوكمة، يجب إعادة موافقة الأعضاء المتأثرين.

إذا كان العضو غير متوافق مع المسار، لا يتحول إلى دافع فعال.

قواعد الكيان

  ↓
قواعد المسار
  ↓

  ↓
شروط العضو

المستوى الأدنى لا يخالف الأعلى إلا إذا سمح الأعلى بالاستثناء.

20. أهم قاعدة في النظام كله
الحقوق والالتزامات لا تُبنى على وجود العضو في الكيان فقط،
بل تُبنى على اشتراكه الفعّال في المسار المناسب داخل المحفظة المناسبة.

يعني:

العضو ← كيان ← محفظة ← مسار ← اشتراك فعّال ← حقوق والتزامات

الخلاصة التقنية

التقنية المطلوبة هي:

منصة ويب/جوال مبنية على محرك قواعد، ومحرك توافق، ومحرك علاقات، ومحرك قرارات، مع دفتر مالي مقسم وسجل تدقيق وطبقة شفافية وخصوصية.

وليست مجرد تطبيق صندوق.

أقوى وصف تقني مختصر:

نظام متعدد الكيانات يدير محافظ اجتماعية قابلة للتخصيص، ويدعم مسارات حوكمة متعددة داخل المحفظة الواحدة، ويربط حقوق العضو والتزاماته بدرجة توافق شروطه مع خصائص الكيان والمحفظة والمسار وبند الصرف، مع دفتر مالي مستقل لكل مسار، وتصويت ومراجعة واعتراض وسجل تدقيق وشفافية قابلة للضبط.

هذا هو العمود الفقري التقني الذي يستوعب كل ما ناقشناه.

نعم. بما أنك لا تريد ASP.NET Core / C#، فأقوى Stack مناسب لهذا المشروع برأيي هو:

الاختيار الأساسي المقترح
TypeScript Full-Stack
Backend

Node.js + NestJS + TypeScript

NestJS مناسب لأن مشروعك يحتاج Modular Architecture واضحة: عضويات، كيانات، محافظ، مسارات حوكمة، تصويت، اعتراضات، دفتر مالي، صلاحيات. وNestJS مبني لـ server-side applications قابلة للتوسع ويستخدم TypeScript رسميًا. 
NestJS Documentation

Database

PostgreSQL

هذا أفضل خيار لأن مشروعك علاقاته كثيرة وحساسة ماليًا. تحتاج معاملات، قيود، فهارس، علاقات، وسجلات مالية قوية. PostgreSQL يدعم Row-Level Security لتقييد الوصول للصفوف حسب السياسات، وهذا مفيد جدًا في الصلاحيات والشفافية حسب العضو/المحفظة/المسار. 
PostgreSQL

ORM

Prisma ORM

مناسب مع TypeScript ويعطيك وصولًا type-safe لقاعدة البيانات ويدعم PostgreSQL، وهذا يقلل الأخطاء في نظام معقد العلاقات. 
Prisma

Frontend Web

Next.js + React + TypeScript

للوحة الإدارة والعضو والمراجع.
ممتاز للواجهات الكبيرة، النماذج، الجداول، التقارير، الصلاحيات، ولوحات التحكم.

Mobile

React Native / Expo

لأن العضو العادي يحتاج جوال: دفع، تصويت، اعتراض، إشعارات، مراجعة اشتراكاته.

TypeScript مناسب هنا لأنه يوحد اللغة بين الواجهة والخلفية.
NestJS يعطيك بنية Modules واضحة.
PostgreSQL يعطيك قوة البيانات.

أهم قواعد تقنية يجب الالتزام بها
1. PostgreSQL هو المصدر الحقيقي

لا تجعل القواعد المالية فقط في الواجهة.
كل عملية مهمة يجب أن تمر من Backend وتُسجل في قاعدة البيانات.

2. لا حذف للعمليات المالية

أي خطأ مالي يعالج بـ:

reversal transaction

adjustment transaction

correction entry

وليس حذفًا.

3. Ledger مستقل

لا تعتمد فقط على جدول payments.

استخدم نموذج:

ledger_accounts
ledger_transactions
ledger_entries
balance_snapshots
